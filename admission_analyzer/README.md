# Анализ конкурсных списков ИТМО

Локальная программа: моделирует зачисление внутри **одного вуза** с учётом приоритетов.

Абитуриент занимает место только на **одной** программе — той, с самым высоким
приоритетом, куда он проходит. Освободившиеся места автоматически переходят
следующим в рейтинге, и так до полной стабилизации.

Поле «Высший проходной приоритет» из исходных файлов **не используется**:
программа пересчитывает этот статус сама, одновременно по всем загруженным
направлениям.

### Без интерфейса

Вся логика лежит в пакете `analyzer` и не зависит от Streamlit:

```python
from analyzer import load_directory, allocate, SCENARIOS
from analyzer.reports import programs_summary

programs, errors = load_directory("data")
registry = {p.key: p for p in programs}
registry["090304 программная инженерия"].seats.budget_total = 150
registry["090304 программная инженерия"].seats.special_quota = 15

result = allocate(registry, SCENARIOS["highest_priority"])
print(programs_summary(result, registry))
```

## Формат входных таблиц

Один файл = одна образовательная программа. Поддерживаются `.tsv`, `.csv`,
`.xlsx`. Кодировки UTF-8, UTF-8 BOM, cp1251 определяются автоматически, как и
разделитель CSV/TSV.

Код направления берётся из имени файла: `090303 прикладная информатика.tsv` →
`09.03.03`.

Ожидаемые столбцы (регистр, пробелы и «ё» не важны — заголовки нормализуются):

| Столбец | Обязателен | Примечание |
|---|---|---|
| Категория | да | протягивается вниз, если указана только у первой строки блока |
| Код | да | идентификатор абитуриента, всегда хранится **строкой** (ведущие нули не теряются) |
| Приоритет | да | целое число |
| Место | нет | место в исходном списке; используется как порядок БВИ |
| Вид испытания | нет | ЕГЭ / ВИ / без испытаний |
| Математика, Информатика, Русский язык | нет | |
| ИД | нет | индивидуальные достижения |
| Балл ВИ, Балл ВИ+ИД | нет | основные критерии сортировки |
| Преимущественное право | нет | |
| Согласие | нет | нужен для строгого сценария |
| Олимпиада | нет | |

Полностью пустые строки и столбцы игнорируются. Если нет `Код`, `Приоритет` или
`Категория`, программа покажет понятную ошибку со списком найденных столбцов.

Распознаваемые категории: БВИ, особая квота, отдельная квота, целевая квота,
общий конкурс, платный конкурс, неизвестная категория. Платный конкурс и
неизвестная категория **не занимают** бюджетных мест.

### Важно: один человек — несколько строк в одной программе

В реальных списках ИТМО один и тот же «Код» встречается в одном файле
**несколько раз**: отдельной строкой в квоте и отдельной строкой в общем
конкурсе. Например, код `1292602` в файле программной инженерии есть и в
целевой квоте (Место 80), и в общем конкурсе (Место 771).

Поэтому единицей данных здесь является **строка**, а не человек, и приоритет
хранится **на строке**: в данных встречаются коды с разными приоритетами на
квотной и общей строке (у той же записи — приоритет 4 в квоте и 1 в общем
конкурсе).

Модель это учитывает: человек может состязаться в нескольких конкурсах одной
программы, но занимает **ровно одно** место — самое «сильное» из доступных
(БВИ → квоты → общий конкурс). Строка в общем конкурсе при этом освобождается
и достаётся следующему.

---

## Сценарии

| Сценарий | Что делает |
|---|---|
| **Строгий текущий** | Учитываются только строки с «Согласие = да». Ближе всего к тому, что происходит «прямо сейчас». |
| **По высшему приоритету** | Учитываются все заявления независимо от согласия. Каждый распределяется на программу с самым высоким приоритетом, куда проходит. Это прогноз «если все дойдут до конца». |
| **Первый приоритет** | Учитываются только заявления с приоритетом 1. |
| **Пользовательский** | Вручную: учитывать ли согласие, БВИ, квоты; перераспределять ли свободные квотные места; максимальный допустимый приоритет; добор квотников в общий конкурс. |

Если в файле **нет** колонки «Согласие», строгий сценарий не выбрасывает всех
его абитуриентов — он учитывает их всех и выдаёт предупреждение.

## Алгоритм распределения

Используется **отложенное принятие** (deferred acceptance, Гейл–Шепли) с
предложениями со стороны абитуриентов:

1. Каждый нераспределённый абитуриент «предлагает себя» программе с самым
   высоким приоритетом, откуда он ещё не был отклонён.
2. Программа рассматривает **всех** накопленных кандидатов заново и оставляет
   лучших, разбирая свои конкурсы по очереди: БВИ → особая квота → отдельная
   квота → целевая квота → общий конкурс. Уже занявший место в более сильном
   конкурсе из последующих исключается.
3. Отклонённые возвращаются в общий пул и идут к следующему приоритету.
   Освободившееся место сразу достаётся следующему в рейтинге.
4. Повторяется, пока никто не меняет положение.

Свойства:

* **Не зависит от порядка файлов и строк.** Рейтинги полностью детерминированы
  (последний критерий — код), а на каждом шаге пул пересматривается целиком.
  Это проверено тестами на всех перестановках строк и на реальных данных.
* **Всегда завершается.** Индекс в списке приоритетов у каждого абитуриента
  только растёт, поэтому число шагов ограничено сверху. Дополнительно стоит
  явный лимит `MAX_ITERATIONS = 10 000`; число итераций логируется и
  показывается в интерфейсе.
* Результат — стабильное паросочетание, оптимальное для абитуриентов.

### Число мест

Вуз не публикует число мест в самих списках, поэтому его задаёт пользователь.
Настройки сохраняются в `programs.json` и подхватываются при следующей загрузке.

Места считаются так:

```
места общего конкурса = всего бюджетных
                      − все квотные места (забронированы)
                      − принятые БВИ            (если «БВИ внутри бюджета»)
                      + незаполненные квотные места (если включено)
```

Квотные места бронируются **первыми**: БВИ их не занимают. Сумма
БВИ + квоты + общий конкурс никогда не превышает общее число бюджетных мест.

### Очерёдность внутри конкурса

Общий конкурс и квоты сортируются так:

1. Балл ВИ+ИД — по убыванию
2. Балл ВИ — по убыванию
3. Математика — по убыванию
4. Информатика — по убыванию
5. Русский язык — по убыванию
6. Наличие преимущественного права
7. Код абитуриента

**БВИ сортируются отдельно и стоят выше обычного конкурса** — по колонке
«Место» из исходного списка, а **не по баллам ЕГЭ**. В данных у строк БВИ баллов
ЕГЭ вообще нет, а «Место» отражает порядок, в котором вуз ранжировал
олимпиадников.

> ### ⚠️ О tie-breaker
>
> Пункт 7 — **код абитуриента** — это **технический** критерий, а не правило
> вуза. Он вынесен в отдельную функцию `analyzer.allocation.tie_breaker` и нужен
> ровно для одного: чтобы при полном равенстве всех официально известных
> критериев результат был воспроизводим и не зависел от порядка строк в файле.
>
> **Реальный порядок вуза при полном равенстве баллов из этих данных
> невосстановим и может отличаться.** Если несколько человек стоят на границе с
> одинаковыми баллами, кто именно пройдёт — эта программа предсказать не может.
> Ориентируйтесь на колонку «Запас / дефицит»: значение около нуля означает, что
> исход не определён.

## Проверки данных

Вкладка «Проверки» показывает:

* одинаковый код дважды внутри одной категории одной программы;
* один код в нескольких категориях (норма для этих данных — помечено как info);
* у одного абитуриента одинаковый приоритет на разных программах;
* пропущенный приоритет;
* неизвестная категория;
* некорректный балл (вне 0–100 для предмета, вне 0–400 для суммы);
* «Балл ВИ» не совпадает с суммой предметов;
* сумма предметов и ИД не совпадает с «Балл ВИ+ИД»;
* код есть только в одном файле;
* количество БВИ больше количества мест;
* сумма квотных мест больше общего числа мест;
* одному коду назначено несколько итоговых программ (ошибка алгоритма);
* в файле отсутствует колонка согласия.

Часть расхождений нормальна: у строк с внутренними испытаниями (ВИ) «Балл ВИ»
не обязан равняться сумме трёх ЕГЭ.

## Отчёт по абитуриенту

Введите свой уникальный код. По каждому его заявлению показывается программа,
приоритет, категория, балл, согласие, место в исходном списке, расчётное место
после распределения, проходит ли он **без учёта** других программ, итоговый
статус, на какую программу распределён, сколько человек впереди, сколько мест и
запас/дефицит до границы.

Отдельно даётся объяснение решения, например:

> Абитуриент не распределён на 09.03.02, потому что проходит на 02.03.03 с приоритетом 2.

> Абитуриент рассматривается на 09.03.03 с приоритетом 2, поскольку на программах
> с более высоким приоритетом не вошёл в число проходящих.

### Цветовые статусы

* 🟢 зелёный — проходит;
* 🟡 жёлтый — пограничная позиция (±5 мест от границы, порог настраивается);
* 🔴 красный — не проходит;
* ⚪ серый — ушёл на программу с более высоким приоритетом.

## Калькулятор по баллам

Если у вас нет кода в загруженных списках (или вы хотите прикинуть «а что если»),
на вкладке «Калькулятор по баллам» можно ввести баллы напрямую — без реального
кода. Добавьте одну строку на каждую программу и приоритет, как при настоящей
подаче документов: программа, категория, приоритет, баллы по предметам и ИД
(либо сразу итоговый балл ВИ+ИД, если знаете его без разбивки). Дальше работает
тот же алгоритм распределения, что и для реальных абитуриентов, — включая
цепные эффекты: введённый профиль может вытеснить реального человека, который,
в свою очередь, займёт место на другой программе.

Технически калькулятор добавляет ваши строки во временную копию затронутых
программ (исходные данные не меняются) под синтетическим кодом и запускает
`allocate()` заново целиком — так учитывается конкуренция именно при вашем
профиле, а не изолированная прикидка «прошёл бы я один».

Для БВИ баллы не нужны, но у гипотетической БВИ-заявки нет «Места» из настоящего
списка олимпиадников — она ставится в конец очереди БВИ, поэтому результат для
БВИ-сценария заведомо пессимистичен.

## Ограничения модели

Это **прогноз по опубликованным спискам**, а не официальный результат.

1. **Число мест задаёт пользователь.** В списках его нет. Ошибка в числе мест
   напрямую двигает проходной балл.
2. **Только один вуз.** Люди уходят в другие вузы, и эти уходы модель не видит —
   реальные проходные баллы обычно оказываются **ниже** прогноза.
3. **Порядок при равенстве баллов технический** (см. выше). На границе прогноз
   ненадёжен.
4. **Данные — срез на момент выгрузки.** Согласия отзывают и подают, приоритеты
   меняют. Файлы устаревают за часы.
5. **Квота → общий конкурс.** В этих данных квотник уже имеет собственную строку
   в общем конкурсе, поэтому модель ничего не достраивает: настройка
   «Добирать квотников в общий конкурс» по умолчанию **выключена**. Включайте её
   только если в ваших файлах такой строки нет, иначе человек будет учтён дважды
   и результат исказится. При включении добираются только те, у кого нет
   собственной строки в общем конкурсе, и только при наличии баллов.
6. **Целевая квота упрощена.** Она моделируется как единый конкурс, тогда как в
   реальности делится по отдельным заказчикам со своими наборами мест.
7. **Правила вуза не реконструируются.** Программа не знает о договорах,
   льготах, особенностях приёма и ручных решениях приёмной комиссии.
8. **Одинаковые приоритеты на разных программах.** Формально так быть не должно;
   если встречается, программа выдаёт предупреждение и упорядочивает такие
   программы по ключу — это тоже технический порядок.

### Почему прогноз не гарантирует зачисление

Модель отвечает на вопрос «как распределились бы **эти** люди по **этим**
спискам при **таком** числе мест, если бы все дошли до конца и никто не ушёл в
другой вуз». Реальное зачисление зависит от оттока в другие вузы, изменений
согласий и приоритетов до дедлайна, точного числа мест и правил вуза при
равенстве баллов. Совпадение прогноза с итогом не гарантировано — используйте
результат для оценки шансов и запаса, а не как решение приёмной комиссии.

## Структура проекта

```
admission_analyzer/
    app.py                    Streamlit-интерфейс (только интерфейс)
    analyzer/
        __init__.py           публичный API
        models.py             доменные модели, категории, статусы, сценарии
        normalization.py      нормализация заголовков, значений, категорий
        loader.py             чтение .tsv/.csv/.xlsx, кодировки, разделители
        allocation.py         рейтинги, tie-breaker, алгоритм распределения
        validation.py         проверки данных и результата
        reports.py            таблицы, объяснения, экспорт
        settings_store.py     programs.json
    data/                     входные файлы
    tests/                    pytest
    requirements.txt
    README.md
```

`validation.py` и `settings_store.py` вынесены из `reports.py` отдельно, чтобы
проверки и хранение настроек не смешивались с формированием отчётов.
